# 文档编写

## 目标

将仓库文档写成符合约定的 VitePress 页面，同时保证目录结构清晰、示例可读、图表与图片资源可维护。

详细规范放在 [references/vitepress-docs.md](references/vitepress-docs.md)。只有在需要确认细节约束时再读取该文件，避免把整份规范都塞进上下文。

## 工作流

### 1. 确认范围

- 仅在用户明确要求新建或修改 `docs/` 文档时执行。
- 先确定目标路径位于 `docs/src/` 下，再判断是新建文档、重写现有文档，还是仅补充局部章节。
- 如果任务来自零散材料，先整理成页面大纲，再落成正式文档。

### 2. 组织文件与资源

- 新建 Markdown 文件时使用 kebab-case 文件名。
- 将 `guide/` 类内容放到 `docs/src/guide/`，包文档按 `docs/src/packages/<pkg>/...` 组织。
- 将 `.drawio` 文件放到当前文档目录下的 `drawio/` 子目录，不要与 `.md` 文件混放。
- 让同一目录层级的多篇文档共享同一个 `drawio/` 目录。

### 3. 整理图片资源

- 当任务涉及 Markdown 图片迁移、重命名、引用改写或未引用图片清理时，按 `$doc-image-maintainer` 的流程执行。
- 目标结构固定为“相对当前 md 文件目录”的 `./imgs`，图片引用统一改为相对路径。
- 对无语义命名的图片使用 `<md-basename>-figure-XX.<ext>` 规则重命名。
- 删除图片前先完成引用改写与二次扫描，避免误删。
- 如果只是编写正文、不涉及现有图片治理，不必主动触发这一流程。

### 4. 编写页面骨架

- 根据需要写 YAML frontmatter；常见字段是 `title` 和 `outline`。
- 让每篇文档都以 `# 标题` 开头，并紧跟一句说明定位和范围的话。
- 默认按以下顺序组织正文：
  1. 整体结构
  2. 核心概念 / 前置知识
  3. 逐文件或逐模块详解
  4. 运行脚本
  5. 相关链接
- 当任务不适合完整顺序时，保留“先解释动机，再展示配置”的写法。

### 5. 写正文与示例

- 正文统一使用简体中文；技术术语、配置项、代码标识符保留英文。
- 给配置或源码示例使用带文件路径的代码块，例如 ` ```ts [src/foo.ts] `。
- 用表格表达参数说明和对比信息；用列表解释步骤、规则和结论。
- 只在“必须特别提醒”的句子里使用带颜色的 `<span>` 强调，不要滥用。
- 将补充说明放进 VitePress 容器：推荐 `tip`、`details`、`warning`、`danger`、`info`。
- 需要目录树时，使用纯文本代码块，并用 `←` 标注关键说明。

### 6. 处理 DrawIO 与脚本块

- 引入 DrawIO 时，不要把 XML 直接内联到 Markdown。
- 使用 `?raw` 导入 `.drawio` 文件，并用 `<ClientOnly>` 包裹 `DrawioViewer`。
- 将 `<script setup>` 放在文档末尾，也就是所有 Markdown 内容之后。
- 当只需确认 DrawIO 引入格式、容器规则、强调样式或章节顺序时，读取 [references/vitepress-docs.md](references/vitepress-docs.md) 对应段落即可。

### 7. 收尾检查

- 新建文档后，补充 `docs/src/.vitepress/configs/sidebar.ts` 对应条目。
- 检查内部链接是否指向正确路径或具体章节。
- 检查代码块路径标注、表格、容器、`<script setup>` 位置是否符合规范。
- 检查 DrawIO 文件命名是否以所属文档名为前缀。
- 如果执行过图片整理，额外确认 `./imgs` 引用、重命名结果和未引用文件清理结果正确。
- 如果只做局部改写，避免顺手重写整篇无关内容。
